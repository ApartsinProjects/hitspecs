Cloud Computing & Cloud-Native Platforms

מחשוב ענן ופלטפורמות ענן-נייטיב

cloud architecture decisions, scalable application patterns, managed compute, storage trade-offs, and production delivery

Design and build scalable cloud applications across compute, storage, and managed services

Year 313 weeks2h lecture + 2h practiceProject-based

About this course

Design and build scalable cloud applications by choosing the right compute, storage, networking, and managed-service patterns for each workload.

Course format. Thirteen weeks, four contact hours each: a two-hour lecture (concepts and theory) and a two-hour practice session. The course is project-based; teams carry one running project end to end and present it three times, in weeks 5, 8, and 13.
What you will build

Designed and engineered a scalable cloud application with an API, background workers, object storage, relational or NoSQL persistence, queues or events, caching, CDN delivery, and containerized or serverless compute, provisioned through Terraform, load-tested with k6, and instrumented with Prometheus and Grafana against latency, reliability, and cost goals.

Expected outcomes

  • Compare cloud compute options: virtual machines, containers, Kubernetes, serverless functions, managed application platforms, and managed databases.
  • Choose appropriate storage for workload requirements: object storage, block storage, file storage, relational databases, document stores, key-value stores, caches, queues, and streams.
  • Design a scalable cloud application around stateless services, durable state, asynchronous workflows, caching, and content delivery.
  • Apply cloud architecture patterns including API gateway, worker queue, event-driven integration, fan-out, materialized views, cache-aside, bulkhead, retry, circuit breaker, and idempotent consumer.
  • Containerize application components and decide when Kubernetes is appropriate versus serverless or managed platform deployment.
  • Model availability, consistency, latency, throughput, and cost trade-offs in cloud architecture decisions.
  • Provision infrastructure declaratively using infrastructure as code and environment-specific configuration.
  • Configure load balancing, autoscaling, and deployment strategies that meet defined latency and availability targets.
  • Instrument services with metrics, logs, traces, alerts, and dashboards tied to service-level objectives.
  • Defend architecture choices using design diagrams, load-test evidence, failure-mode analysis, and cost estimates.

Key topics

  • Compute options
  • Cloud storage choices
  • Scalable application design
  • Cloud architecture patterns

Theoretical foundations

The concepts and results this course rests on.

  • Cloud compute placement: virtual machines, containers, Kubernetes, serverless functions, and managed application platforms
  • Cloud storage semantics: object, block, file, relational, document, key-value, cache, queue, and stream systems
  • CAP theorem and the availability, consistency, partition-tolerance trade-off
  • Replication, partitioning, indexing, caching, and eventual-consistency models for cloud data
  • Stateless service design, idempotency, backpressure, retries, timeouts, and circuit breakers
  • Reconciliation and the declarative desired-state control loop
  • Queueing theory and tail-latency analysis under load
  • Autoscaling control theory and capacity versus cost models
  • Supply-chain provenance, reproducible-build theory, and deployment safety

Prerequisites

This is a Year-3 course. It assumes the mandatory CS core: data structures and algorithms, operating systems, computer networks, databases, software engineering, and the core mathematics (linear algebra, probability and statistics, calculus, discrete mathematics). It additionally requires the specific prior courses listed below.

Course-specific prerequisites:

  • Operating systems and computer networks
  • Software engineering and a programming language

AI tools in this course.

Students use AI assistants to compare cloud architecture options, draft architecture decision records, scaffold Terraform modules, generate Dockerfiles, sketch API and worker boundaries, and produce candidate storage schemas for relational, document, key-value, cache, queue, and object-storage services. They use chat-based and editor-integrated tools to generate k6 load scripts and realistic traffic, then connect agents to kubectl and cloud MCP servers to inspect deployment state and propose fixes. AI also helps interpret Prometheus and Grafana output, summarizing latency, queue depth, cache hit ratio, and cost data into concrete autoscaling and design actions. Teams are expected to review every generated architecture and manifest critically, since a plausible but wrong compute or storage choice can create a system that cannot scale.

Student project

Teams design and build one scalable cloud application from workload requirements through architecture decision records, then implement an API, durable storage, object storage, asynchronous workers, caching or CDN delivery, and a deployment path using containers, serverless functions, or managed platforms. The system is provisioned through infrastructure as code, load-tested, instrumented, and defended with evidence for scalability, reliability, and cost.

Requirements

  • Build a working system, not a set of disconnected exercises.
  • Be original: a new system that solves a real problem, not a re-implementation of a tutorial or course demo.
  • Show real depth: real data, real users or realistic load, and engineering trade-offs that are measured rather than assumed.
  • Carry one running project from specification to a deployed, defensible result across the whole term.
  • Work in a team of three or four and defend the design at each of the three presentations (weeks 5, 8, and 13).

Tools & platforms

  • Docker: build and run application containers
  • Kubernetes: orchestrate containerized workloads
  • Helm: package and template Kubernetes manifests
  • Istio: manage service to service traffic and policy
  • Terraform: provision cloud infrastructure as code
  • AWS Lambda: implement serverless functions
  • Amazon S3: provide object storage for files, logs, and static assets
  • Amazon RDS: provide managed relational database storage
  • Amazon DynamoDB: provide managed key-value and document storage
  • Amazon SQS and Amazon SNS: provide durable queues and pub-sub messaging
  • Amazon API Gateway: expose managed HTTP APIs
  • Amazon CloudFront: deliver content through a CDN
  • Redis: provide low-latency caching and rate-limit state
  • PostgreSQL: model relational cloud application data
  • Prometheus: collect cluster and application metrics
  • Grafana: visualize metrics and build dashboards
  • Amazon Web Services: host managed cloud services
  • kubectl: operate clusters from the command line
  • Trivy: scan container images for vulnerabilities
  • k6: run load and performance tests
